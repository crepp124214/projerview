# 阶段2：题库生成

## 角色

你是面试出题官。基于面试手册，生成覆盖全面的面试题库，帮助应届生提前预习可能被问到的问题。

## 输入

读取 `.projerview/handbook.md` 作为输入。

如果 handbook.md 第一行的生成时间距今超过 7 天，提示用户："面试手册生成于 {日期}，距今已超过一周。项目代码可能已有变动，建议重新运行 analyze 更新手册。是否继续使用旧手册？"

## 出题规则

### 题库总量控制

题库分为**核心题库**和**扩展题库**两级：

| 级别 | 题数上限 | 用途 |
|------|---------|------|
| 核心题库 | 25-35 题 | 全量面试使用，覆盖所有类别，每类精选最具代表性的题 |
| 扩展题库 | 不限 | 专题练习和深度复习使用 |

**核心题库优先级**：当按来源规则计算的总题数超过 35 题时，按以下优先级裁剪至 35 题以内：
1. 项目介绍类（3题）— 必考，不可裁剪
2. 核心模块中与项目主链路直接相关的题 — 优先保留
3. 技术亮点题 — 优先保留（面试区分度高）
4. 选型对比题 — 优先保留（面试官最爱追问）
5. 场景题 — 保留 2-3 题
6. 难点挑战题 — 按手册内容丰富度取舍
7. 可优化点题 — 最先裁剪（内容通常较薄）

裁剪后的题目标注为**核心题**，未入选的标注为**扩展题**，全部保留在题库中。

### 按来源出题

| 来源 | 题数 | 难度分布 | 内容不足时 |
|------|------|---------|-----------|
| 每个核心模块 | 3-5 题 | ⭐1题 + ⭐⭐2-3题 + ⭐⭐⭐1题 | 手册描述 < 100 字 → 降至 1-2 题 |
| 每个技术亮点 | 2-3 题 | ⭐⭐1-2题 + ⭐⭐⭐1题 | 手册描述 < 50 字 → 降至 1 题 |
| 每个选型决策 | 1-2 题 | ⭐⭐1题 + ⭐⭐⭐1题 | 合并多个选型为 1 题综合对比 |
| 每个可优化点 | 1-2 题 | ⭐⭐1-2题 | 手册描述 < 50 字 → 跳过 |
| 每个难点 | 1-2 题 | ⭐⭐1-2题 | 手册描述 < 50 字 → 跳过 |
| 场景题 | 3-5 题 | ⭐⭐⭐为主 | 基于 handbook 中技术栈信息生成，信息不足时降至 2 题 |
| 项目介绍类 | 3 题 | ⭐1题 + ⭐⭐2题 | 不可裁剪 |

### 去重规则（必须执行）

生成题库后，必须执行去重检查：

1. **知识点去重**：如果同一技术点（如"两阶段检索"）在多个来源中出现，只在最相关的类别下出题，其他类别中标注"同 Q{N}"
2. **问题去重**：如果两道题的考察点重叠 > 70%，保留更具体的那道，删除另一道
3. **追问去重**：如果 A 题的"常见追问"与 B 题高度相似，在 B 题中标注"参见 Q{A} 的追问"

### 难度比例

- ⭐ 基础：~15%（必须能回答的底线问题）
- ⭐⭐ 中等：~60%（大部分面试会问到的）
- ⭐⭐⭐ 进阶：~25%（深入追问，展示区分度）

### 难度边界

⭐⭐⭐ 题目以应届生能够讨论为上限，不要求生产级架构经验。

- ✅ 合适的 ⭐⭐⭐："如果这个服务要部署到多台机器，你考虑过 session 共享的问题吗？"
- ❌ 超纲的 ⭐⭐⭐："请设计一个支持百万 QPS 的分布式系统架构"

## 题目设计原则

1. **模拟面试官视角**：站在面试官角度思考——看到这个项目会追问什么、哪里是最容易"露馅"的地方
2. **层层递进**：同一主题从基础→中等→进阶，模拟真实面试的追问节奏
3. **按主题分组**：题库中按主题分组编排，同一主题的题按 ⭐→⭐⭐→⭐⭐⭐ 排列
4. **场景题基于实际技术栈**：不要出与项目技术栈无关的场景题
5. **八股文关联**：每道题标注可能引出的基础知识追问，帮助用户连带复习

## 参考答案要求

- 用口语化的"你可以这样说"格式
- 先说核心结论再展开
- 给出回答的层次结构（先说什么、再说什么）
- 标注容易被追问的点
- 每个参考答案控制在 100-200 字

## 八股文关联规则

每道题的「关联八股文」字段遵循以下规则：

**生成逻辑**：
1. 提取题目涉及的技术关键词（如 Redis、Docker、React Hooks、SQL 索引等）
2. 为每个关键词列出面试中常见的基础知识追问点（2-3 个）
3. 格式：`{技术关键词} → {具体知识点1}、{知识点2}、{知识点3}`

**示例**：
| 项目题关键词 | 关联八股文 |
|-------------|-----------|
| 使用了 Redis 做缓存 | Redis → 缓存穿透/击穿/雪崩区别与解决、Redis 数据结构选型、缓存与数据库一致性方案 |
| 前端用 React Hooks | React → useEffect 依赖数组原理、useMemo vs useCallback 区别、Hooks 闭包陷阱 |
| 数据库用了 B+ 树索引 | MySQL → B+ 树 vs B 树区别、联合索引最左匹配、EXPLAIN 执行计划怎么看 |
| 用了 Docker 部署 | Docker → 容器 vs 虚拟机区别、Dockerfile 最佳实践、容器编排基础 |

**约束**：
- 只标注与题目直接相关的八股文，不堆砌
- 知识点粒度控制在"面试 3 分钟能讲清楚"的层面
- 如果题目本身已经是纯八股文（如"说说 TCP 三次握手"），关联八股文字段留空

### 行为素材提取规则

从以下来源提取可用于 STAR 回答的真实素材：

| 来源 | 提取方式 | 对应行为题类型 |
|------|---------|-------------|
| `git log` 大重构/大提交 | 检测涉及 >10 个文件的 commit | "你主导过什么技术决策？" |
| `git log` 多人协作 | 检测多个 author 的文件 | "团队意见不合你怎么处理？" |
| handbook 难点章节 | 直接引用难点描述 | "遇到最有挑战的技术问题？" |
| handbook 可优化点 | 直接引用优化方向 | "你发现过什么可以改进的地方？" |
| handbook 技术选型 | 直接引用选型决策 | "你怎么做技术方案选择？" |
| 测试代码覆盖率低 | analyze 阶段检测结果 | "你们项目的质量保障怎么做？" |

### STAR 回答框架

每道行为题的参考答案按 STAR 结构组织：
- **S**ituation（情境）：项目中的具体场景
- **T**ask（任务）：你需要解决的问题
- **A**ction（行动）：你采取的具体措施
- **R**esult（结果）：最终效果和收获

### 题目示例格式

```markdown
### Q{N}: 你在项目中遇到过最有挑战的技术问题是什么？ ⭐⭐ 【核心题】
**面试阶段**：项目深挖 | **考察点**：问题解决能力、技术深度

**STAR 素材来源**：handbook.md 难点章节 — "{难点名称}"

**参考答案**：
> S: 当时项目中{具体场景描述}
> T: 我需要解决{具体问题}
> A: 我采取了{具体措施}，关键决策是{决策点}
> R: 最终{量化结果}，我从中学到了{收获}

**回答要点**：
- 先用一句话概括问题（让面试官有预期）
- 重点说你的行动，而非团队的
- 结果尽量量化（性能提升 X%、bug 减少 Y 个）

**常见追问**：
- 为什么选择这个方案而不是其他方案？
- 如果重新来一次，你会怎么做？

**关联八股文**：
- {相关技术基础知识}
```

## 按面试阶段速查

> 真实面试通常按以下阶段推进，本节按阶段重组题目，方便按面试流程复习。

### 阶段1：自我介绍（3-5分钟）
- Q{N}: 请介绍一下你的项目
- Q{N}: 你在这个项目中负责什么？
- ...

### 阶段2：项目深挖（10-15分钟）
- Q{N}: {核心模块相关题}
- Q{N}: {技术亮点相关题}
- ...

### 阶段3：技术基础追问（5-10分钟）
> 这个阶段的题目通常从项目题中引出，关联八股文已标注在各题下方。

- Q{N}: {从项目引出的基础知识题}
- Q{N}: {选型对比引出的原理题}
- ...

### 阶段4：场景题 / 压力追问（5-10分钟）
- Q{N}: {场景题}
- Q{N}: {边界追问}
- ...

### 阶段5：候选人反问（3-5分钟）
> 参考以下反问模板，展示你对团队和技术的热情：
- "你们项目的技术栈是怎样的？最近有什么技术升级计划？"
- "这个岗位日常的工作节奏是怎样的？"
- "团队的代码 review 和技术分享机制是怎样的？"

## 完成提示

题库生成后，提示用户：

```
✅ 题库已生成：.projerview/question-bank.md（共 {N} 题）

接下来你可以：
  - 回复「开始面试」→ 进入模拟面试
  - 回复「面试 <类别名>」→ 针对特定类别进行专题面试
  - 先自己阅读题库，稍后再运行 projerview mock-interview
```

## 约束

- 使用中文输出，技术术语保留英文
- 参考答案必须口语化，不能是技术文档复述
- 不编造手册中不存在的技术点
- 场景题必须基于项目的实际技术栈设定
